## 坑 1：API 调用太频繁，被限流了

**场景**: 刚启动系统，14 个 Agent 一起调用数据 API
**结果**: 5 分钟后，API 返回 429（Too Many Requests）
**心情**: 从兴奋到绝望😰

**解决方案**:
加了个本地缓存层，同样的数据不重复请求。
**效果**: API 调用量降了 60%，再也不怕限流了！

```python
# 伪代码，大概就这个意思
@cache(ttl=300)  # 缓存 5 分钟
def get_stock_data(symbol):
    return api.get(symbol)
```

---

## 坑 2：Agent 之间"失联"，各干各的

**场景**: Dev Agent 修好了 bug，但其他 Agent 不知道
**结果**: 重复劳动，效率极低
**心情**: 我这是搭了个啥…🤦‍♂️

**解决方案**:
引入 PM Agent（项目经理），所有任务都通过它分配。
每个 Agent 完成任务后，向 PM 汇报，PM 再通知其他 Agent。

**效果**: 团队终于像团队了！

---

## 坑 3：没做风控，差点"裸奔"实盘

**场景**: 系统跑通了，兴奋得想马上实盘
**结果**: 被朋友拦住了："你风控呢？"
**心情**: 后背发凉…😰

**解决方案**:
紧急加了三层风控：
1. **Guard Agent** - 每笔交易二次验证
2. **Risk Agent** - 单票不超过 10%
3. **Ops Agent** - 异常立即告警

**效果**: 现在系统安全多了，虽然还没实盘，但心里有底了。

---

## 坑 4：技术栈选太复杂，把自己绕晕了

**场景**: 一开始想用 Kafka + Redis + PostgreSQL + InfluxDB…
**结果**: 光环境配置就搞了一天
**心情**: 我是来炒股还是来运维的？😤

**解决方案**:
简化！简化！再简化！
- 数据库：只用 SQLite（后期可换 PostgreSQL）
- 消息队列：用 Redis Streams（轻量）
- 任务调度：APScheduler（够用就行）

**效果**: 系统轻量了，维护也简单了。

## 我的技术栈（最终版）

**任务调度**: APScheduler

## 几条真心建议

### 1️⃣ 从小处着手
别一上来就想搞个完美的系统。
先跑通最小闭环：数据→策略→交易→记录
然后再慢慢优化。

### 2️⃣ 重视风控
宁可错过机会，不要做错交易。
风控不是可有可无，是必须的！

### 3️⃣ 先模拟，后实盘
我打算模拟跑 3 个月再考虑实盘。
别急，市场永远在那里。

### 4️⃣ 持续迭代
没有完美的系统，只有更好的系统。
我每天都在改代码，每天都有新优化。
